# Implementace i1 · Fundament — načtení, zobrazení, spec fixes

> **Pozice v roadmapě:** iterace 1 z 5 · blokuje i2–i5
> **Rozsah:** první funkční vertikální řez od DB přes core až do React plot panelu
> **Navazuje na:** v0.4 §5–9, §10.1–10.2, virtual_test §7.1 (č. 1–6)
> **Cíl:** uživatel otevře bank, vybere notu a vidí křivky všech pěti parametrů s korektními metadaty. Nic víc.

Tato iterace řeší **spec debt před F2 startem** (6 položek z virtual_test §7.1) a postaví
minimální read-only pipeline. Žádné operátory, žádné anchors, žádné úpravy — jen
spolehlivý průchod dat. Cokoliv dalšího (i2–i5) předpokládá, že i1 je stabilní.

## 1. User stories

### US-1.1 · Uživatel načte bank

> *„Jako zvukař chci v UI vybrat `bank_001` ze seznamu a do 500 ms vidět
> metadata (jméno, instrument, počet not, datum importu), abych věděl, že
> pracuji nad správnou verzí."*

**Acceptance:**
- `GET /api/banks` vrátí seznam s `{id, name, instrument, n_notes, created_at, parent_id}`
- `GET /api/banks/:id` vrátí kompletní metadata včetně `provenance` (import source, timestamp)
- FE komponenta `<BankSwitcher>` v headeru renderuje dropdown, `<BankMetaPanel>` detail
- Cache hit (in-process LRU) < 50 ms; cache miss (SQLite JSON load + ICR parse) < 500 ms pro bank s 88 × 8 notami

### US-1.2 · Uživatel vybere notu a vidí partialy

> *„Potřebuji vybrat `(m=60, v=5)` a dostat tabulku všech naměřených partialů
> s jejich `k`, `f_hz`, `A₀`, `τ₁`, `τ₂`, `a₁`, `beat_hz`, `phi`,
> `fit_quality`, `sigma` a `origin`. Sloupec `origin` mi řekne, jestli hodnota
> přišla z měření, z extrapolace, nebo z anchoru."*

**Acceptance:**
- `GET /api/banks/:id/notes/:midi/:velocity` vrátí `NoteDetail` s polem `partials`
- Každý partial nese i nově zavedené pole `origin ∈ {measured, derived, extrapolated, anchored}` a `sigma` (viz §3)
- FE komponenta `<PartialTable>` (right rail) renderuje tabulku s filtrem `fit_quality ≥ threshold`
- Tabulka se updatuje reaktivně přes TanStack Query při změně `noteId`

### US-1.3 · Uživatel vidí křivky pěti parametrů

> *„Chci plot s osou k (1..60) a hodnotou parametru, pro každý ze čtyř zvolených
> parametrů. Log-space toggle. Fit quality jako barva bodu. Over-hover na
> bodu ukáže `k`, přesnou hodnotu a sigmu."*

**Acceptance:**
- `<PlotCanvas>` (Plotly) zobrazí 1–5 křivek (user toggle checkboxes: τ₁, τ₂, A₀, a₁, β/f)
- Pozitivní parametry (τ₁, τ₂, A₀) v log-y, ostatní lineárně
- Body obarveny podle `fit_quality` (continuous colormap 0→1, viridis)
- Tooltip: `k · param · value ± σ · origin · quality`
- Přepnutí noty (US-1.2) triggeruje re-render plotu do 100 ms (data už v cache)

### 2.1 SIGMA_0 default hodnoty

Odvozeno z math §8.1 (σ₀ = variance raw observable při q=0.5) a **empirické
kalibrace** proti 5 referenčním bankám z `idea/` (viz `scripts/calibrate_sigma0.py`,
229 766 partial pairs z 10 bank×bank kombinací). Hodnoty konfigurovatelné
per-bank přes `bank.metadata.sigma_0_override`:

```python
# piano_core/constants.py
SIGMA_0: Final[dict[str, float]] = {
    # All values EMPIRICAL (MAD of pairwise differences / sqrt(2))
    "tau1":    0.174,    # sekundy — spec heuristika 0.06 byla ~3x podcenená
    "tau2":    0.534,    # sekundy — spec heuristika 0.08 byla ~7x podcenená
    "A0":      0.075,    # spec heuristika 0.12 byla mírně nadsazená
    "a1":      0.452,    # p84-of-differing (bimodal: ~80% partialů má a1=1.0)
                         # spec heuristika 0.05 dramaticky podcenila reálný rozptyl
    "f_coef":  0.003,    # dimensionless residual f_hz/(k·f0·sqrt(1+B·k²)) - 1
                         # přesná shoda se spec heuristikou
    "beat_hz": 0.23,     # spec heuristika 0.30 byla mírně nadsazená
}
```

**Kalibrační proces** (reprodukovatelný):
```bash
python scripts/calibrate_sigma0.py
```
Skript auto-discovery všech `idea/*.json` v ICR formátu, spočítá 10 pairwise
kombinací, robustní MAD estimátor s p84 fallbackem pro bimodální parametry.

**Reference banky použité při kalibraci:**
- `as-blackgrand-04162340-raw-icr.json` (AS Black Grand, k_max=128)
- `ks-grand-2604161547-icr.json` (ks-grand, k_max=80)
- `ks-grand-04130018-raw.json` (k_max=60)
- `icr-bank-sample1.json`, `icr-bank-sample2.json` (k_max=60, sibling processings)

**Revalidace:** při přidání dalších banků rerun skriptu. Současná kalibrace je
zdravá — žádný parametr nemá >90% identických párů napříč poolem.

## 3. Backend

### 3.1 Core model update — `Partial`

```python
# piano_core/models/partial.py
from dataclasses import dataclass
from typing import Literal

Origin = Literal["measured", "derived", "extrapolated", "anchored"]

@dataclass(frozen=True, slots=True)
class Partial:
    k: int
    f_hz: float                           # absolutní (Hz); f_coef = f_hz/(k·f0·√(1+B·k²)) je derived
    A0: float
    tau1: float
    tau2: float
    a1: float
    beat_hz: float
    phi: float
    fit_quality: float
    # NEW v i1:
    sigma: float | None = None            # F-2 — per-partial measurement uncertainty
    origin: Origin = "measured"           # F-3
```

Migrace existujících bank v JSON blobu: ICR reader dosadí `sigma=None, origin="measured"`
pro záznamy bez těchto polí. Writer vždy zapisuje novou verzi schématu `icr_version=2`.

### 3.2 Operator base — `ApplyResult`

```python
# piano_core/operators/base.py
@dataclass(frozen=True)
class ApplyResult:
    bank: Bank
    edit: EditRecord
    diagnostics: ApplyDiagnostics

class Operator(ABC):
    @abstractmethod
    def apply(self, bank: Bank, params: P) -> ApplyResult: ...
```

Všechny čtyři existující/plánované operátory (`SplineTransfer`, `PhysicalExpand`,
`ConsensusMerge`, `AnchorInterpolate`) v i2–i3 musí tuto signaturu dodržet.

### 3.3 Repository + router

Pro i1 stačí tenké čtecí endpointy:

| Metoda | Path | Handler |
|---|---|---|
| `GET` | `/api/banks` | `BankRepository.list()` → serialize summary |
| `GET` | `/api/banks/:id` | `BankRepository.load()` + metadata |
| `GET` | `/api/banks/:id/notes` | seznam `(midi, velocity)` tuples |
| `GET` | `/api/banks/:id/notes/:midi/:velocity` | `Note` → `NoteDetail` (s partialy) |
| `GET` | `/api/banks/:id/notes/:midi/:velocity/curves` | `CurvesPayload` (pre-computed pro plot) |

`CurvesPayload` schéma:
```python
class CurvesPayload(BaseModel):
    midi: int
    velocity: int
    parameters: dict[str, list[CurvePoint]]   # "tau1" → [CurvePoint, ...]

class CurvePoint(BaseModel):
    k: int
    value: float
    sigma: float | None
    fit_quality: float
    origin: Origin
```

Pre-computing na serveru šetří FE logiku a umožní pozdější agresivnější caching.

## 5. Database

SQLite schéma pro i1 se nemění, pouze rozšiřuje ICR JSON schéma. SQL migrace:

```sql
-- Preferovaná cesta (CREATE TABLE — STORED generated column funguje):
CREATE TABLE IF NOT EXISTS banks (
    id          TEXT PRIMARY KEY,
    data_json   TEXT NOT NULL,                              -- JSON blob s ICR payloadem
    parent_id   TEXT REFERENCES banks(id),
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    instrument  TEXT GENERATED ALWAYS AS
                (json_extract(data_json, '$.metadata.instrument')) STORED
);
CREATE INDEX IF NOT EXISTS idx_banks_instrument ON banks (instrument);

-- Pokud tabulka už existuje (SQLite nepodporuje ADD COLUMN se STORED, pouze VIRTUAL):
ALTER TABLE banks ADD COLUMN instrument TEXT
  GENERATED ALWAYS AS (json_extract(data_json, '$.metadata.instrument')) VIRTUAL;
CREATE INDEX IF NOT EXISTS idx_banks_instrument ON banks (instrument);
```

**Poznámky k SQLite:**
- `data_json` je `TEXT` s JSON obsahem; dotazy používají `json_extract()` / `json_tree()` (JSON1 je součást stdlib buildu)
- Generated columns s `STORED` vyžadují SQLite 3.31+ (2020) a musí být definovány při `CREATE TABLE`
- Foreign keys se aktivují přes `PRAGMA foreign_keys = ON` na každém connection
- Pro write concurrency zapnout `PRAGMA journal_mode = WAL`

Žádné nové tabulky. Anchors, edits, cache jsou mimo rozsah i1.

## 8. Risks & open questions

- **R-1:** ICR v1 → v2 migrace na produkčních datech — potřebujeme dry-run skript
  a rollback plán. Řeší DevOps v rámci i1 ve sprint-2.
- **R-2:** `sigma` field backfill pro stará data je `None`, což v i2 znamená,
  že BLUE consensus nemá per-partial varianci z měření a musí spadnout na
  `fit_quality`-derived proxy. Explicitně zdokumentováno v API schématu.
- **R-3:** `origin = "anchored"` v i1 nemůže nikdy vzniknout (anchors jsou až v i2).
  Enum to povoluje, ale validator v writer cestě musí ověřit konzistenci s existencí
  Anchor záznamu. Implementace až v i2.
