# Implementace i4 · Stiffness UI, objevování matematických souvislostí

> **Pozice v roadmapě:** iterace 4 z 5 · vyžaduje i1–i3, blokuje i5
> **Rozsah:** interaktivní stiffness ovládání (weight anchors s vizuální zpětnou vazbou),
> diagnostic view matematických fitů (B, τ-power, A₀ envelope), relationship discovery
> (γ = τ₂/τ₁, cross-note monotonicity, physical model residuals)
> **Navazuje na:** math §3 (physical priors jako hypothesis), math §8 (Bayesian update),
> v0.4 §3 (state store), tone_identification_and_correction.md §2.2 (IdentificationProvenance)
> **Cíl:** pokročilý uživatel (akustik/výzkumník) objevuje strukturu dat — kde leží
> fyzikální limity nástroje, jestli se bank chová konzistentně napříč klaviaturou, a jak
> pevné ukotvení odpovídá jeho intuici.

i3 odevzdala core editační workflow. i4 otáčí orientaci: od „oprav data" k „poznej data".
Stiffness slider je zobrazení Kalman update v reálném čase; relationship discovery
zkoumá, jestli bank respektuje fyziku, kterou si myslíme, že by měl.

## 1. User stories

### US-4.1 · Uživatel nastavuje stiffness s vizuální zpětnou vazbou

> *„Mám anchor `(k=15, τ₁, 0.445)` s výchozím `weight=0.5`. Tahám slider
> v rozsahu 0–1 a během tažení vidím spline deformovat: při `weight=0`
> prochází „vedle" anchoru, při `weight=1` prochází anchorem přesně. Pod
> splinem se zároveň updatuje uncertainty band. Slider má tři checkpointy
> se štítky: `soft` (0.3), `default` (0.5), `hard` (1.0)."*

**Acceptance:**
- `<StiffnessSlider>` komponenta s debounced preview (150 ms) během drag
- Preview volá `/api/ops/anchor-interpolate` s `commit=false` — single round-trip per checkpoint
- Uncertainty band vykreslen jako Plotly `fill=toself` polygon okolo splinu
- Po release slideru se commituje aktuální hodnota (standardní anchor PATCH z i2)

### US-4.2 · Uživatel vidí Kalman update vizualizaci

> *„V preview módu chci vidět rozklad: původní spline (modrá), anchor bod
> (oranžový romb s chybovou úsečkou), výsledný spline po update (zelená).
> Malý popover vysvětluje vzorec a vyčíslené `K = σ²_cons / (σ²_cons + σ²_a)`."*

**Acceptance:**
- `<KalmanUpdateExplainer>` komponenta: tři trace v jednom plotu + formula LaTeX přes KaTeX
- Hodnota `K` dynamicky aktualizovaná při změně `weight`
- Toggle `advanced view` v `OperatorDrawer` — default skryté, pro power userů zapnuté

### US-4.3 · Uživatel objevuje matematické vztahy

> *„Spustím `Discover Relationships` na bank. Dostanu report:
>   - Inharmonicity `B` napříč notami: hodnoty a fit quality per nota.
>   - Power-law exponent α pro `τ₁(k) ∝ k^(-α)`: histogram + outliers.
>   - Ratio γ = τ₂/τ₁: per nota, ukazuje systematický trend?
>   - A₀ envelope parametry (β, μ) napříč velocity.
>   - Cross-note monotonicity: pro každý `(velocity, k, parameter)` ověří, jestli
>     hodnoty monotónně klesají s rostoucím midi (fyzikální předpoklad)."*

**Acceptance:**
- `GET /api/banks/:id/math-analysis` vrací `MathAnalysisReport`
- `<MathRelationshipsPanel>` v novém tabu vedle AnomalyList
- Klik na outlier notu → scroll do NotePicker + předvyplnění do drawer

### US-4.4 · Uživatel vidí provenance detaily

> *„U každé noty, která byla upravena operátorem, chci rozkliknout
> `IdentificationProvenance`: seznam zdrojových banků (hash), anchor IDs
> (resolve na human-readable), fitnuté `B_hat ± σ`, `α̂ ± σ`, GCV-optimální
> λ, random_seed."*

**Acceptance:**
- `<ProvenanceDetail>` v `<EditHistoryList>` per záznam
- Payload v `EditRecord.diagnostics.identification_provenance` (už existuje z i3)
- Formátování LaTeX odhadů s chybami; hash zdrojového banku jako tooltip s full ID

### US-4.5 · Uživatel porovná notu s fyzikálním priorem

> *„Jeden klik v note detail: `Compare with physical prior`. Plot překreslí
> naměřené body + křivku fitnutou přes `PhysicalExpand`-style power-law.
> Tabulka ukazuje RMSE, max odchylku, partialy s residuálem > 2σ."*

**Acceptance:**
- `POST /api/ops/physical-expand` s `commit=false` → diagnostics obsahuje residuals
- `<PhysicalPriorOverlay>` tracer na PlotCanvas (toggleable)
- Residual table jako collapsible pod plotem

### 2.1 Komponentní hierarchie

```
<AnchorEditorPopover>            # z i2, rozšíření v i4
├── <AnchorMetaForm>              # k, parameter, value
├── <StiffnessSlider>             // NEW
│   ├── <SliderTrack>

│   └── <CurrentValueBadge>
├── <KalmanUpdateExplainer>       // NEW (advanced mode)
│   ├── <MiniPlot>                 // tri-trace: prior, anchor, posterior
│   ├── <FormulaLatex>             // LaTeX via KaTeX
│   └── <KValueBadge>              // K = σ²c / (σ²c + σ²a)
└── <CommitBar>
```

### 2.2 Slider semantika

`weight ∈ [0, 1]` → `σ_a = (1 - weight) · SIGMA_0[parameter]`.

- `weight = 0` → σ_a = σ₀ (max uncertainty, anchor má minimální vliv)
- `weight = 0.5` → σ_a = 0.5·σ₀ (default)
- `weight = 1.0` → σ_a → 0 (hard anchor, force hard_replace v Phase B per i3)

Slider má logaritmický mapping na `σ_a` (lidsky příjemnější než lineární) —
převod přes `sigma_a = SIGMA_0[param] * (1 - weight)**2`.

### 2.3 Preview volání

```ts
const previewMutation = useAnchorInterpolate({ commit: false });

const onSliderChange = useMemo(() => debounce((w: number) => {
  previewMutation.mutate({
    target_note_ids: [[midi, velocity]],
    parameters: [anchor.parameter],
    anchor_overrides: { [anchor.id]: { weight: w } },   // speciální path in i4
    random_seed: 42,   // fixed during preview for stable visuals
  });
}, 150), [anchor.id]);
```

Backend endpoint si při `anchor_overrides` vytvoří ephemeral Bank s upravenou váhou,
spustí interpolaci, ale neuloží.

## 3. Math Analysis Service

### 3.1 Backend modul

```
piano_core/analysis/
├── __init__.py
├── cross_note.py       # monotonicity, smooth trends přes MIDI
├── physical_fit.py     # B, α, β, μ per note
├── relationships.py    # γ ratio, derived correlations
└── outlier_detect.py   # LOF / MAD na keyboard-wide distribuci
```

### 3.2 `MathAnalysisReport` schema

```python
class MathAnalysisReport(BaseModel):
    bank_id: str
    generated_at: datetime

    # Per note diagnostics
    per_note: dict[str, NoteMathDiag]    # "midi_velocity" → diag

    # Keyboard-wide aggregates
    inharmonicity_trend: TrendAnalysis        # B(midi) — očekává roustoucí
    tau1_power_trend: TrendAnalysis           # α(midi) — relativně stabilní
    a0_envelope_trends: dict[str, TrendAnalysis]   # β(midi), μ(midi)
    gamma_ratio_stats: RatioStats              # τ₂/τ₁ napříč všemi notami

    # Violations
    monotonicity_violations: list[MonotonicityViolation]
    outlier_notes: list[OutlierNote]

class NoteMathDiag(BaseModel):
    midi: int
    velocity: int
    B_hat: float
    B_sigma: float
    tau1_alpha: float          # τ₁ ∝ k^(-α)
    tau1_alpha_sigma: float
    A0_beta: float             # log A₀ = c - β log k - μ k
    A0_mu: float
    gamma: float               # median τ₂/τ₁
    fit_rmse: dict[str, float] # per parameter
    physical_prior_fit_quality: float  # overall score 0-1

class MonotonicityViolation(BaseModel):
    parameter: str
    velocity: int
    k: int
    midi_from: int
    midi_to: int                  # sousední note páry
    expected_direction: Literal["decreasing", "increasing"]
    delta: float                  # pozorovaná změna
    severity: Literal["minor", "major"]
```

### 3.3 Implementace — cross-note monotonicity

Pro každou trojici `(velocity, k, parameter)` (kromě `f_coef`, který je per-note fyzikální)
seřaď podle midi, ověř očekávaný směr:

| Parametr | Očekávaný směr růstu midi |
|---|---|
| `τ₁` | klesá (vyšší tóny decayují rychleji) |
| `τ₂` | klesá |
| `A₀` | klesá |
| `a₁` | monotonie nespolehlivá — jen hledáme extrém outliers |
| `B` (inharmonicity) | roste (fyzika tuhé struny) |

Violation = více než 2 po sobě jdoucí páry porušují směr s rozdílem > 0.1·σ.

### 3.4 API

| Metoda | Path | Účel |
|---|---|---|
| `GET` | `/api/banks/:id/math-analysis` | Plný report (cache 10 min) |

## 5. WebSocket progress protocol (rozšíření v0.4 §12)

Math analysis a long-running operace emitují progress events. i4 přidává:

```typescript
// WS message types
type OperatorProgress = {
  type: "operator.progress";
  op_id: string;
  status: "running" | "done" | "error";
  progress: number;       // 0-100
  eta_seconds: number | null;
  current_step: string;   // "fitting inharmonicity on note 60..."
};
```

Frontend `<ProgressToast>` renderuje aktivní operations v rohu. ESC cancels (propojí
s backend `AbortController` per op_id).

## 6. Service layer rozšíření

### 6.1 `AnalysisService.math_analysis`

```python
class AnalysisService:
    async def math_analysis(
        self, bank_id: str, op_id: str | None = None,
    ) -> MathAnalysisReport:
        bank = await self._load(bank_id)
        n_notes = len(bank.notes)

        per_note = {}
        for i, note in enumerate(bank.notes):
            if op_id:
                await self.ws.emit(op_id, {
                    "progress": int(100 * i / n_notes),
                    "current_step": f"fitting note {note.midi}/{note.velocity}",
                })
            per_note[f"{note.midi}_{note.velocity}"] = fit_physical(note)

        aggregates = compute_aggregates(per_note)
        violations = find_monotonicity_violations(bank)
        outliers = detect_outlier_notes(per_note, threshold=3.0)

        if op_id:
            await self.ws.emit(op_id, {"progress": 100, "status": "done"})
        return MathAnalysisReport(...)
```

### 6.2 Cache & invalidace

- Key: `math:{bank_id}`
- TTL: 10 min
- Invalidace: na `bank.invalidated` pro `bank_id`

## 8. Akceptační kritéria i4

1. Stiffness slider je interaktivně plynulý (60 fps drag na referenčním HW)
2. Kalman explainer formule a hodnoty odpovídají kódu (snapshot test s ručně spočítaným příkladem)
3. Math analysis na celém banku (88×8 = 704 not) doběhne do 8 s P95 s progress eventy
4. Report CSV export obsahuje všechna pole `NoteMathDiag` + aggregates
5. Monotonicity detection má FP rate < 5% na synthetic „clean" bank datasetu
6. ProgressToast správně spojuje notifikace z více paralelních ops

## 9. Risks & open questions

- **R-12:** Math analysis je compute-heavy (704 NLS fits na bank). Pro progress events
  potřebujeme jemnější granularitu než per-note (např. per-velocity-group). Mitigace:
  ProcessPool executor, 4 workers, shared progress counter přes `multiprocessing.Value`.
- **R-13:** `γ = τ₂/τ₁` výpočet může mít dělení nulou pokud `τ₁` degenerovaná. V edge
  case math §11 už je handling, ale na agregaci přes klávesnici potřebujeme robust
  statistic (median místo mean, trimmed IQR).
- **R-14:** Stiffness preview round-trip (2× round-trip na checkpoint) může na pomalém
  spojení cítit jako lag. Mitigace: klientská aproximace (fast local spline recompute)
  pro intermediate values, server-authoritative jen při release. Odloženo na v1.0 — v i4
  ponecháváme server round-trip, protože cache hit je rychlý.
- **R-15:** Monotonicity expectations (§3.3 tabulka) jsou empirical rules of thumb.
  Některé piano restorations mají záměrně netypický dekadenční profil. Možné false
  positives. Řešeno flagem `severity` + user dismiss.
- **R-16:** Math analysis report je velký payload (stovky per-note entries). Pro
  malé banky OK, ale u production-scale dat je načítání pomalé. Mitigace: streaming
  response (NDJSON), lazy loading per section. V i4 zatím full JSON.
