# Implementace i2 · Matematická vrstva + anchors a editace splinů

> **Pozice v roadmapě:** iterace 2 z 5 · vyžaduje i1, blokuje i3–i5
> **Rozsah:** `piano_core/completion/` a `piano_core/splines/` moduly + první interaktivní
> editační flow na FE (anchor CRUD, spline recompute s preview)
> **Navazuje na:** `math_partial_completion.md` §3–§9, §11–§12, v0.4 §7 (Operator pattern)
> **Cíl:** uživatel vidí existující anchors na křivce, přidá/smaže/upraví anchor a v preview vidí
> přepočítanou křivku s uncertainty band. Stále pouze jedna nota, jeden parametr; přenosy a
> operátory na celé bance jsou až v i3.

Klíčová změna oproti i1: do systému vstupuje **zdroj pravdy o tvaru křivky** — matematická
pipeline, která z observations + anchors vyrobí estimate + uncertainty. Všechny další iterace
tuto pipeline konzumují.

## 1. User stories

### US-2.1 · Uživatel vidí existující anchors

> *„Když otevřu notu, která má anchors, chci je vidět přímo na plotu jako odlišné
> body (kosočtverec místo kolečka) s popiskem `weight` a metainformací `origin =
> anchored`. Chci je mít i v samostatném panelu `<AnchorList>` seřazené podle k."*

**Acceptance:**
- `GET /api/banks/:id/notes/:m/:v/anchors` vrátí všechny anchors pro notu napříč parametry
- `<PlotCanvas>` renderuje anchors jako druhý scatter trace (odlišný symbol, offset z-order nad data)
- `<AnchorList>` v right rail ukazuje `k · parameter · value · weight · created_at · created_by`

### US-2.2 · Uživatel přidá nový anchor

> *„Kliknu na bod křivky v plotu, objeví se popover s formulářem: `k`
> (prefilled), `parameter` (prefilled), `value` (prefilled z kliknutí),
> `weight` (slider 0–1, default 0.5). Potvrzení vytvoří anchor a spline se
> okamžitě přepočítá. Vidím side-by-side před/po s color-coded diffem."*

**Acceptance:**
- `POST /api/banks/:id/notes/:m/:v/anchors` s payloadem `AnchorCreate`
- FE mutation invaliduje `['note', ...]`, `['curves', ...]`, `['anchors', ...]`
- Preview mode: před commitem se volá spline recompute endpoint s `commit=false`
  a zobrazuje se dry-run výstup v semi-transparent overlay nad původní křivkou

### US-2.3 · Uživatel upraví weight (stiffness) anchoru

> *„Mám anchor s `weight=0.3`, přetáhnu slider na `0.9` a chci okamžitě vidět,
> jak se spline víc přimkne k anchoru. Reálně zpoždění pod 200 ms aby to
> fungovalo interaktivně."*

**Acceptance:**
- `PATCH /api/banks/:id/anchors/:aid` s body `{weight: 0.9}` (žádná další pole)
- Debounce 150 ms na slider change, během drag se zobrazuje preview bez persistence
- Při release se commituje (PATCH + invalidate)
- Backend hodí `400` pokud `weight ∉ [0, 1]`

### US-2.4 · Uživatel smaže anchor

> *„Klik na anchor v listu nebo v plotu → context menu → Delete. Spline se
> přepočítá, undo dostupný v EditHistoryCrumbs."*

**Acceptance:**
- `DELETE /api/banks/:id/anchors/:aid`
- Undo opakovaným POST se stejnými parametry v 30s okně (idempotency key z FE)

### US-2.5 · Uživatel přepočítá spline s anchors (ruční override)

> *„Chci pro vybraný parametr spustit AnchorInterpolate s nastaveným
> `prior_weight` a `smoothing` (slider 0–1, None = GCV automat). Vidím
> preview a můžu commitnout jako nový Bank."*

**Acceptance:**
- `POST /api/ops/anchor-interpolate` s `AnchorInterpolateParams` (viz v0.4 §7.5)
- Vrací `ApplyDiagnostics` s GCV-selected λ, PCHIP fallback flagem, per-k `sigma`
- Commit persistuje nový Bank, non-commit vrací jen diff

## 2. Matematická vrstva — moduly

Struktura pod `piano_core/completion/` a `piano_core/splines/`. Každý modul je pure-function
layer — žádný IO, žádná globální konfigurace, vše přes argumenty. Všechny funkce přijímají
`random_seed: int` (F-4 z i1) a vrací deterministický výstup.

### 2.2 `physical_priors.py` — math §3

Tři funkce, všechny čisté:

```python
def fit_inharmonicity_nls(
    partials: Sequence[Partial],
) -> tuple[float, float]:
    """
    Fit B z f_coef = sqrt(1 + B·k²).
    Returns (B_hat, B_var). Implementace: scipy.optimize.curve_fit.
    Edge case math §11.3: pokud var(f_coef - 1) < 1e-6 → return (0.0, 1e-12).
    """

def wls_log_log(
    y: np.ndarray, k: np.ndarray, w: np.ndarray,
) -> tuple[np.ndarray, np.ndarray]:
    """
    log y = c - α log k. Vrací (phi=[c, α], Sigma=2×2 cov matrix).
    Používá se pro τ₁, τ₂ power-law extrapolaci.
    """

def wls_amplitude_envelope(
    partials: Sequence[Partial],
) -> tuple[np.ndarray, np.ndarray]:
    """
    log A₀ = c - β log k - μ k. Multi-variate WLS (math §3.4).
    """
```

Každá funkce má docstring-linked test vektor z `math_partial_completion.md` §3
se snapshot tolerancí 1e-4.

### 2.3 `anchor_interpolate.py` — math §12.2

Kanonická unified pipeline:

```python
def anchor_interpolate(
    partials: Sequence[Partial],
    anchors: Sequence[Anchor],
    parameter: Literal["tau1", "tau2", "A0", "a1", "f_coef"],   # f_coef = derived inharmonicity residual
    *,
    prior_weight: float = 0.3,
    smoothing: float | None = None,   # None → GCV
    random_seed: int = 0,
) -> InterpolationResult:
    """
    Returns:
      InterpolationResult(
        estimate: Callable[[int], float],   # θ̂(k)
        sigma:    Callable[[int], float],   # σ_θ̂(k)
        lambda_used: float,
        used_pchip: bool,                   # §6.4 monotonicity fallback
        coverage: tuple[int, int],          # (k_min_observable, k_max_observable)
      )
    """
```

Vnitřně volá:

1. `splines.log_space.transform` pokud parameter ∈ {τ₁, τ₂, A₀}
2. `splines.penalized.fit` s GCV grid search (math §4.2.1)
3. `splines.pchip.override` pokud monotonicity violation (§6.4)
4. `completion.uncertainty.bootstrap` s `n_boot=200` a `random_seed`

## 3. Anchor datový model a persistence

### 3.1 Core `Anchor`

```python
@dataclass(frozen=True, slots=True)
class Anchor:
    id: str                                   # uuid4
    midi: int
    velocity: int
    k: int
    parameter: Literal["tau1", "tau2", "A0", "a1", "f_coef"]
    value: float
    weight: float                             # ∈ [0, 1]; 1.0 = hard
    origin: Literal["manual", "imported", "regression_fit"]
    created_at: datetime
    created_by: str                           # user id or "system"
    note: str = ""                            # free-text uživatelská poznámka
```

### 3.2 Bank integrace

```python
@dataclass(frozen=True, slots=True)
class Bank:
    # ...
    anchors: tuple[Anchor, ...] = ()

    def anchors_for_note(self, midi: int, velocity: int) -> tuple[Anchor, ...]:
        return tuple(a for a in self.anchors if a.midi == midi and a.velocity == velocity)

    def with_added_anchor(self, a: Anchor) -> "Bank":
        return replace(self, id=new_id(), parent_id=self.id, anchors=self.anchors + (a,))
```

Anchor CRUD vytváří novou verzi Bank (immutable chain). Optimalizace: anchor-only
mutace neklonuje pole `notes` do nové struktury, sdílí referenci na tuple.

### 3.3 DB schéma (rozšíření v0.4 §9)

```sql
CREATE TABLE anchors (
    id          TEXT PRIMARY KEY,                    -- UUID v4 jako string
    bank_id     TEXT NOT NULL REFERENCES banks(id),
    midi        INTEGER NOT NULL,
    velocity    INTEGER NOT NULL,
    k           INTEGER NOT NULL,
    parameter   TEXT NOT NULL,
    value       REAL NOT NULL,                       -- IEEE 754 double
    weight      REAL NOT NULL CHECK (weight BETWEEN 0 AND 1),
    origin      TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),  -- ISO 8601 UTC
    created_by  TEXT NOT NULL,
    note        TEXT NOT NULL DEFAULT ''
);
CREATE INDEX idx_anchors_bank_note ON anchors (bank_id, midi, velocity);
CREATE INDEX idx_anchors_bank_param ON anchors (bank_id, parameter);
```

Uložení mimo JSON blob (na rozdíl od notes) proto, že anchors chceme často dotazovat
cross-bank (v i3 pro propagaci napříč notami).

## 6. Testy

### 6.1 Math layer — property-based (hypothesis)

- `test_anchor_interpolate_is_deterministic_with_seed`
- `test_hard_anchor_weight_1_passes_through_value` (hodnota musí být ≤ 1e-9 od anchor.value)
- `test_pchip_fallback_preserves_monotonicity` (generator: decreasing sequences)
- `test_gcv_selects_lambda_minimizing_cv_error` (grid search correctness)
- `test_bootstrap_variance_scales_with_sample_size`

### 6.2 Edge cases (math §11)

- Konflikt anchors na stejném `(k, parameter)` s `|v₁ - v₂| > 3σ₀` → operator raises
  `multiple-anchor-conflict`
- B ≈ 0 detekce → return `(0.0, 1e-12)` namísto NLS divergence
- Extreme extrapolation guard (`Δk > 2·|K_obs|`) → `is_physically_estimated=True` in diag

### 6.3 API integration

- `test_post_anchor_creates_new_bank_version` (parent_id správný)
- `test_patch_anchor_weight_triggers_ws_broadcast`
- `test_delete_anchor_is_reversible_within_grace_window` (undo path)
- `test_anchor_interpolate_preview_does_not_persist`

### 6.4 FE e2e (3 paths)

- Create → adjust weight → delete → verify history
- Preview → cancel (žádný nový Bank)
- Preview → commit → verify new bank selected in switcher

## 8. Risks & open questions

- **R-4:** GCV grid search může vybrat suboptimální λ na noise-heavy datech. Mitigace:
  po GCV validovat RMSE na held-out subsetu, pokud divergentní → warning do diagnostics.
- **R-5:** Bootstrap uncertainty je O(n_boot · n_k) a pro n_boot=200 může být na
  hot path pomalá. Alternativa: analytical variance z penalty matice (rychlejší, mírně
  méně přesná). V i2 ponecháváme bootstrap, optimalizaci přes caching per-note.
- **R-6:** Hard anchor handling — v Kalman update (math §8.2) s σ_a = 0 vzniká
  numerická nestabilita. Řešení `virtual_test §7.1 #2`: přepis kroku 3 fáze A na
  **globální refit přes augmented dataset** (math §8.3). V i2 implementováno přímo
  v `anchor_interpolate` (anchors jako pseudo-observations s vysokou vahou), Kalman
  path použít jen pro soft anchors.
- **R-7:** Konkurentní edit anchorů dvěma klienty na stejné notě → optimistic
  concurrency konflikt (parent_id se rozchází). Řešení: FE prezentuje conflict dialog,
  server odmítá `409 Conflict` pokud `If-Match: parent_id` neodpovídá.
